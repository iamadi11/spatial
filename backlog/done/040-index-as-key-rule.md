## PM Plan

**Problem**: Using an array index as a React `key` prop (`key={i}`) is one of the most common and subtle React anti-patterns. When the list order changes (insert, delete, reorder), index-based keys cause React to incorrectly reuse component instances — leading to stale state, missed animations, and unnecessary re-renders. The engine can detect missing keys (`missing-key-prop`, rule 036), but has no rule for the index-as-key pattern where keys ARE present but are numeric.

**Goal**: Add an `index-as-key` rule that flags same-type sibling groups where all children have key props AND all key values are non-negative integer strings (e.g., `"0"`, `"1"`, `"2"`), indicating index-based keys generated by `key={i}`.

**Scope**:
- Node-level rule: inspect `node.children` groups of same type
- For each group with 2+ children where ALL have `props.key`:
  - Check if every key value is a string that parses as a non-negative integer
  - If yes → fire warning on the parent node
- Skip groups where any child is missing `props.key` (that's `missing-key-prop`'s job)
- Skip groups where any key is non-numeric (e.g., stable IDs like `"user-123"`)

**Non-goals**: Detecting whether the index keys will cause actual reorder bugs, analyzing runtime list operations, tracking key value changes across renders

**Done when**:
- Rule fires when a parent has 2+ same-type children all with numeric string keys
- Rule returns `triggered: false` when keys are non-numeric stable IDs
- Rule returns `triggered: false` when children are missing keys (not this rule's concern)
- Rule returns `triggered: false` for single-child or different-type sibling groups
- Tests cover happy/edge/failure/unknown

## QA Test Plan

**Happy path 1**: Same-type children with keys "0","1","2" → fires, rule=`index-as-key`.
**Happy path 2**: Non-consecutive numeric keys ("0","5","10") → fires.
**Edge case 1**: Stable string ID keys ("user-1","user-2") → does not fire.
**Edge case 2**: Children without key props at all → does not fire (missing-key-prop handles this).
**Edge case 3**: Different types with numeric keys → does not fire (groups independently).
**Edge case 4**: Single child of a type → does not fire (group < 2).
**Edge case 5**: Mixed keys (some numeric, some not) in same group → does not fire.
**Failure case**: No children → does not fire.
**Failure case 2**: Negative integer strings ("-1","-2") → does not fire.

## Implementation Plan

- `isNonNegativeIntegerString(value)`: regex `^\d+$` check — only digit strings pass
- `detect(node, _metrics)`: group children by type → for each group ≥ 2 where all have `props.key` → check all keys are non-negative integer strings → fire
- O(n) over children per node, O(1) per child key check. Pure function, no DOM.
